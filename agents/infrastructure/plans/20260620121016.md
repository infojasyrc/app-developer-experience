## Architecture Overview

```
Internet
  │
  └─ ALB (app-alb, public, HTTPS 443) ──WAF──► SG: sg-app-alb
       ├── api.<domain>  → TG: api    → ECS service: cm-api    (port 3000)
       └── app.<domain>  → TG: webapp → ECS service: cm-webapp  (port 3000)

VPC-internal
  └─ ALB (tools-alb, internal, HTTPS 443) ──► SG: sg-tools-alb
       ├── unleash.<domain>  → TG: unleash  → ECS service: cm-tools (port 4242)
       ├── grafana.<domain>  → TG: grafana  → ECS service: cm-tools (port 3000)
       └── keycloak.<domain> → TG: keycloak → ECS service: cm-tools (port 8080)

ECS Cluster: appdevexp-dev (shared, Container Insights enabled)
  ├── Service: cm-api      — API container (NestJS, port 3000)
  │                           + MongoDB sidecar (port 27017)
  │                           ← EFS access point: /mongodb-data
  │
  ├── Service: cm-webapp   — Webapp container (Next.js, port 3000)
  │
  └── Service: cm-tools    — Unleash (4242) + Grafana (3000) + Keycloak (8080)
                              + PostgreSQL sidecar (port 5432)
                              ← EFS access point: /postgres-data

EFS File System: appdevexp-dev-efs (encrypted, KMS)
  ├── Access point: /mongodb-data  (uid/gid 999 — mongodb user)
  └── Access point: /postgres-data (uid/gid 70  — postgres user)

Secrets Manager:
  /appdevexp/dev/mongo/admin       — MongoDB root credentials
  /appdevexp/dev/keycloak/admin    — Keycloak admin credentials
  /appdevexp/dev/postgres/tools    — PostgreSQL password for tools DB
  /appdevexp/dev/unleash/api-key   — Unleash API key (set post-bootstrap)
```

**Key design decisions:**
- One ECS cluster, **three ECS services**: `cm-api`, `cm-webapp`, `cm-tools` (isolation by service + security group).
- MongoDB and PostgreSQL run **as ECS sidecar containers** with EFS volumes for persistent data — no managed DB services needed.
- Tools ALB is **internal** (VPC-only); zero internet exposure for Unleash, Grafana, Keycloak.
- All secrets injected via Secrets Manager `secrets` block in task definitions.
- All resources carry `project = var.application_name` for `make list-resources`.
- `ci_cm_components.yml` handles image build + ECS service update; `deploy_cm_infrastructure.yml` handles only Terraform.

## Why Containerized Databases for a Learning Application

| Aspect | Managed (DocumentDB + RDS) | Containerized (EFS-backed) |
|---|---|---|
| Monthly cost | ~$69/month | ~$5/month |
| Setup complexity | Low (Terraform managed) | Medium (EFS mount + access points) |
| HA / failover | Automatic | None (acceptable for learning) |
| Backup | Automated snapshots | AWS Backup for EFS (optional) |
| Data persistence | Fully managed | EFS survives task restarts/replacements |
| Learning value | Low — AWS does everything | High — you manage the full stack |

**How EFS ensures data survives container restarts:**  
The EFS volume is mounted into the container at the data directory (e.g., `/data/db` for MongoDB).
When a Fargate task stops and restarts — whether from a crash, deployment, or scaling event — the new
task mounts the same EFS access point and finds all data intact. EFS is not in-memory; it is a
durable network file system backed by AWS infrastructure.

**Caveat:** MongoDB and PostgreSQL containers should run with `desired_count = 1` and **no auto-scaling**
to avoid two tasks writing to the same EFS path simultaneously (neither supports concurrent multi-writer
access). For a learning app this is acceptable.

## Phase 2 — EFS Module (Persistent Storage for Containerized Databases)

> **Goal:** Create an EFS file system with access points for MongoDB and PostgreSQL data.
> ECS tasks mount access points at container startup. One PR.

### Step 2-1: Create `cloud/terraform/aws/module/efs/`

**`module/efs/main.tf`:**

```hcl
# ── Security Group ────────────────────────────────────────────────────────────
# NFS port 2049 open only to ECS task security groups

resource "aws_security_group" "efs" {
  name        = "${var.application_name}-efs-sg"
  description = "EFS mount targets: NFS from ECS task SGs only"
  vpc_id      = var.vpc_id

  ingress {
    description     = "NFS from app tasks"
    from_port       = 2049
    to_port         = 2049
    protocol        = "tcp"
    security_groups = var.allowed_sg_ids
  }

  egress {
    description = "No outbound from EFS"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = []
  }

  tags = merge(var.tags, { Name = "sg-efs" })
}

# ── File System ───────────────────────────────────────────────────────────────

resource "aws_efs_file_system" "main" {
  creation_token   = "${var.application_name}-efs"
  encrypted        = true
  kms_key_id       = var.kms_key_arn
  performance_mode = "generalPurpose"
  throughput_mode  = "bursting"

  lifecycle_policy {
    transition_to_ia = "AFTER_30_DAYS"
  }

  tags = merge(var.tags, { Name = "efs-${var.application_name}" })
}

# ── Mount Targets (one per private subnet) ───────────────────────────────────

resource "aws_efs_mount_target" "main" {
  count           = length(var.private_subnets)
  file_system_id  = aws_efs_file_system.main.id
  subnet_id       = var.private_subnets[count.index]
  security_groups = [aws_security_group.efs.id]
}

# ── Access Points ─────────────────────────────────────────────────────────────

resource "aws_efs_access_point" "mongodb" {
  file_system_id = aws_efs_file_system.main.id

  root_directory {
    path = "/mongodb-data"
    creation_info {
      owner_uid   = 999   # mongodb user inside mongo:7.0 image
      owner_gid   = 999
      permissions = "755"
    }
  }

  posix_user {
    uid = 999
    gid = 999
  }

  tags = merge(var.tags, { Name = "efs-ap-mongodb" })
}

resource "aws_efs_access_point" "postgres" {
  file_system_id = aws_efs_file_system.main.id

  root_directory {
    path = "/postgres-data"
    creation_info {
      owner_uid   = 70    # postgres user inside postgres:16 image
      owner_gid   = 70
      permissions = "700"
    }
  }

  posix_user {
    uid = 70
    gid = 70
  }

  tags = merge(var.tags, { Name = "efs-ap-postgres" })
}
```

Single ECS task definition with four containers:

| Container | Image | Port | EFS Volume |
|---|---|---|---|
| `postgres` | `postgres:16-alpine` | 5432 | `/postgres-data` (EFS AP) |
| `unleash` | `unleashorg/unleash-server:6` | 4242 | none |
| `grafana` | `grafana/grafana:11.0.0` | 3000 | none |
| `keycloak` | `quay.io/keycloak/keycloak:25.0` | 8080 | none |

The `postgres` container is the sidecar that Unleash and Keycloak both connect to via `localhost`
(they share the same task network namespace in awsvpc mode — all containers in a task share
the local loopback interface).

## Phase 6 — CI/CD: Enable ECS Deploy in `ci_cm_components.yml`

> **Goal:** The existing `ci_cm_components.yml` already builds and pushes images to ECR.
> The `ecs-deploy` composite action step is disabled with `if: false` in both jobs.
> This phase enables it by aligning env vars with Terraform-created resource names
> and removing the `if: false` condition.

### Context: What the existing pipeline already does

The `ci_cm_components.yml` workflow:
1. Detects which packages changed (via `.github/actions/get-changed-packages`)
2. For each changed service: builds the Docker image and pushes to ECR
3. Calls `.github/actions/ecs-deploy` — **currently disabled** (`if: false`)

The ECS deploy composite action updates the ECS service by registering a new task definition
revision with the new image URI, then calls `aws ecs update-service` to trigger a rolling
deployment. The service waits for stability if `wait-for-service-stability: "true"`.

### Step 6-1: Align ECS env vars with Terraform resource names

The current env vars in the workflow are placeholders that don't match what Terraform creates.
The correct names (from `module/ecs-app` outputs and `module/cluster`) are:

| Env Var | Current value (placeholder) | Correct value (from Terraform) |
|---|---|---|
| `ECS_CLUSTER` | `cluster-app` | `appdevexp-dev` |
| `ECS_API_SERVICE` | `conference-api` | `appdevexp-dev-api` |
| `ECS_WEB_SERVICE` | `conference-webapp` | `appdevexp-dev-webapp` |
| `ECS_TASK_DEFINITION` | `frontend-image` | `appdevexp-dev-api` (or `appdevexp-dev-webapp` per job) |
| `API_CONTAINER_NAME` | `ms-conference-api` | `conference-manager_api` |
| `WEB_CONTAINER_NAME` | `ms-conference-webapp` | `conference-manager_webapp` |
