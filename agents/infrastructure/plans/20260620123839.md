## Architecture Overview

```
Internet
  │
  └─ ALB (webapp-alb, public, HTTPS 443) ──WAF──► sg-webapp-alb
       └── app.<domain> → TG: webapp → ECS service: cm-webapp (port 3000)
                               │
                               │ server-side fetch from Next.js container
                               │ (never from the browser — browser never touches the API directly)
                               ▼
VPC-internal
  └─ ALB (api-alb, internal) ──► sg-api-alb  [ingress: sg-webapp-tasks only]
       └── → TG: api → ECS service: cm-api (port 3000)
                          └── MongoDB sidecar (localhost:27017, EFS-backed)

VPC-internal
  └─ ALB (tools-alb, internal) ──► sg-tools-alb  [ingress: VPC CIDR only]
       ├── → TG: unleash    → cm-tools (port 4242)
       ├── → TG: grafana    → cm-tools (port 3000)
       ├── → TG: prometheus → cm-tools (port 9090)
       └── → TG: keycloak  → cm-tools (port 8080)
              └── PostgreSQL sidecar (localhost:5432, EFS-backed)

ECS Cluster: appdevexp-dev
  ├── Service: cm-webapp (Next.js, port 3000) — sg-webapp-tasks
  ├── Service: cm-api    (NestJS + MongoDB sidecar, port 3000) — sg-api-tasks
  └── Service: cm-tools  (Unleash + Grafana + Prometheus + Keycloak + PG sidecar) — sg-tools-tasks

EFS File System: appdevexp-dev-efs (encrypted, KMS)
  ├── Access point: /mongodb-data  (uid 999, gid 999 — mongo:8.0-noble)
  └── Access point: /postgres-data (uid 70, gid 70 — postgres:18-alpine)
```

**Why the API is private and the webapp proxies API calls:**

Next.js runs server-side on the ECS container. When the webapp needs data, Next.js server
code calls the internal API ALB directly using a VPC-private DNS name. The browser receives
rendered HTML or JSON — it never calls the API endpoint directly. This means:

- `NEXT_PUBLIC_*` environment variables for the API URL are **not needed** — the call is
  server-to-server inside the VPC, not browser-to-server across the internet.
- The API security group can reject every connection that does not originate from the
  webapp task security group. Zero attack surface from the public internet.
- If any endpoint requires client-side fetching (e.g., SWR, React Query), the webapp exposes
  it as a Next.js API route (`/api/...`) which proxies to the internal API — the browser
  still only ever talks to `app.<domain>`.

**Key invariants:**
- One ECS cluster, three ECS services, three load balancers (1 external, 2 internal).
- Only the webapp ALB is internet-facing. The API ALB and tools ALB are internal.
- All ECS tasks run in **private subnets** with `assign_public_ip = false`.
- All secrets injected via Secrets Manager `secrets` block — never in environment variables.
- All resources tagged `project = "appdevexp"` for `make list-resources`.

### Step 0-3: Create Secrets Manager entries — Makefile approach

#### Why a Makefile, not ad-hoc CLI commands

The `aws secretsmanager create-secret` commands from Rev 2 were raw CLI one-liners.
The problem: if you lose the command, forget the exact secret path, or need to set up a
second environment, you have to reconstruct it from memory. A Makefile target solves all
of these problems:

- **Discoverable:** `make help` shows it alongside other bootstrap targets.
- **Reproducible:** same command, same result, regardless of who runs it.
- **Auditable:** the secret paths and key names are committed to the repo (the values are not).
- **Safe:** passwords are never passed as shell arguments — they would appear in `ps` output and
  shell history. Instead they are read from a gitignored `.env.secrets` file.

#### Who runs it and when

| Question | Answer |
|---|---|
| Who? | The AWS account owner or admin — the person who has `secretsmanager:CreateSecret` permission. This is NOT the `appdevexp-deployer` OIDC role (which is limited to Terraform operations). |
| When? | Once per environment, before the first `terraform apply`. Never repeated unless the environment is rebuilt from scratch or a secret is rotated (use `secrets-rotate` target for that). |
| From where? | The operator's local machine using their admin AWS credentials profile. |

#### Step 0-3-A: Create `cloud/terraform/aws/makefiles/aws-secrets.mk`

```makefile
# ==============================================================================
# AWS Secrets Manager bootstrap
#
# One-time, admin-only targets for creating application secrets.
# Passwords are read from $(SECRETS_ENV_FILE) — NEVER passed as CLI arguments.
#
# Run order: secrets-check → secrets-bootstrap
# To rotate a secret:  secrets-rotate SECRET_NAME=<name>
# ==============================================================================

SECRETS_ENV_FILE ?= .env.secrets

.PHONY: secrets-check secrets-bootstrap secrets-rotate secrets-verify secrets-help

secrets-check: ## 🔍 Verify .env.secrets exists and has required variables
	@if [ ! -f "$(SECRETS_ENV_FILE)" ]; then \
		echo ""; \
		echo "ERROR: $(SECRETS_ENV_FILE) not found."; \
		echo "Copy .env.secrets.example and fill in the values:"; \
		echo "  cp .env.secrets.example $(SECRETS_ENV_FILE)"; \
		echo "  vim $(SECRETS_ENV_FILE)"; \
		echo ""; \
		exit 1; \
	fi
	@for var in MONGO_ADMIN_USERNAME MONGO_ADMIN_PASSWORD \
	            KEYCLOAK_ADMIN_USERNAME KEYCLOAK_ADMIN_PASSWORD \
	            POSTGRES_TOOLS_PASSWORD \
	            GRAFANA_ADMIN_PASSWORD; do \
		if ! grep -q "^$$var=" $(SECRETS_ENV_FILE); then \
			echo "ERROR: $$var is missing from $(SECRETS_ENV_FILE)"; \
			exit 1; \
		fi; \
	done
	@echo "✅ $(SECRETS_ENV_FILE) looks complete."

secrets-bootstrap: secrets-check ## 🔐 Create all Secrets Manager entries (one-time, admin)
	@echo "Creating secrets in region $(AWS_REGION)..."
	@set -a && source $(SECRETS_ENV_FILE) && set +a && \
	aws secretsmanager create-secret \
		--name "/appdevexp/dev/mongo/admin" \
		--description "MongoDB admin credentials for conference-manager dev" \
		--secret-string "{\"username\":\"$$MONGO_ADMIN_USERNAME\",\"password\":\"$$MONGO_ADMIN_PASSWORD\"}" \
		--region $(AWS_REGION) \
		--tags Key=project,Value=appdevexp Key=environment,Value=dev \
		2>/dev/null && echo "✅ Created /appdevexp/dev/mongo/admin" \
		|| echo "⚠️  /appdevexp/dev/mongo/admin already exists — skipping"
	@set -a && source $(SECRETS_ENV_FILE) && set +a && \
	aws secretsmanager create-secret \
		--name "/appdevexp/dev/keycloak/admin" \
		--description "Keycloak bootstrap admin credentials for conference-manager dev" \
		--secret-string "{\"username\":\"$$KEYCLOAK_ADMIN_USERNAME\",\"password\":\"$$KEYCLOAK_ADMIN_PASSWORD\"}" \
		--region $(AWS_REGION) \
		--tags Key=project,Value=appdevexp Key=environment,Value=dev \
		2>/dev/null && echo "✅ Created /appdevexp/dev/keycloak/admin" \
		|| echo "⚠️  /appdevexp/dev/keycloak/admin already exists — skipping"
	@set -a && source $(SECRETS_ENV_FILE) && set +a && \
	aws secretsmanager create-secret \
		--name "/appdevexp/dev/postgres/tools" \
		--description "PostgreSQL password for tools (Unleash + Keycloak + Grafana) in dev" \
		--secret-string "{\"password\":\"$$POSTGRES_TOOLS_PASSWORD\"}" \
		--region $(AWS_REGION) \
		--tags Key=project,Value=appdevexp Key=environment,Value=dev \
		2>/dev/null && echo "✅ Created /appdevexp/dev/postgres/tools" \
		|| echo "⚠️  /appdevexp/dev/postgres/tools already exists — skipping"
	@set -a && source $(SECRETS_ENV_FILE) && set +a && \
	aws secretsmanager create-secret \
		--name "/appdevexp/dev/grafana/admin" \
		--description "Grafana admin password for conference-manager dev" \
		--secret-string "{\"adminPassword\":\"$$GRAFANA_ADMIN_PASSWORD\"}" \
		--region $(AWS_REGION) \
		--tags Key=project,Value=appdevexp Key=environment,Value=dev \
		2>/dev/null && echo "✅ Created /appdevexp/dev/grafana/admin" \
		|| echo "⚠️  /appdevexp/dev/grafana/admin already exists — skipping"
	@echo ""
	@echo "✅ Secrets bootstrap complete. Run 'make secrets-verify' to confirm."

secrets-rotate: ## 🔄 Rotate a single secret. Usage: make secrets-rotate SECRET_NAME=/appdevexp/dev/mongo/admin
	@if [ -z "$(SECRET_NAME)" ]; then echo "ERROR: SECRET_NAME is required"; exit 1; fi
	@set -a && source $(SECRETS_ENV_FILE) && set +a && \
	aws secretsmanager rotate-secret \
		--secret-id "$(SECRET_NAME)" \
		--region $(AWS_REGION)
	@echo "✅ Rotation initiated for $(SECRET_NAME)"

secrets-verify: ## ✅ List secret ARNs (values not shown) to confirm they exist
	@echo "=== Secrets in /appdevexp/dev/ ==="
	@aws secretsmanager list-secrets \
		--filter Key=name,Values=/appdevexp/dev/ \
		--region $(AWS_REGION) \
		--query 'SecretList[].{Name:Name,ARN:ARN,LastChanged:LastChangedDate}' \
		--output table

secrets-help: ## ✨ Show secrets management commands
	@echo ""
	@echo "Secrets Management — one-time admin setup"
	@echo ""
	@echo "  make secrets-check      Verify .env.secrets is complete"
	@echo "  make secrets-bootstrap  Create all secrets in Secrets Manager"
	@echo "  make secrets-verify     List existing secrets (values not shown)"
	@echo "  make secrets-rotate SECRET_NAME=<path>  Rotate one secret"
	@echo ""
	@echo "Required file (gitignored): $(SECRETS_ENV_FILE)"
	@echo "Template file (committed):  .env.secrets.example"
```

### Step 0-4: `terraform.tfvars` — What to commit and what to keep out of git

#### The problem

`terraform.tfvars` is tracked by git. Any value placed in it is visible to anyone with
repo access, stored in git history forever, and potentially logged in CI output.

#### Rule: Sensitivity classification

| Variable | Example | Sensitive? | Where it lives |
|---|---|---|---|
| `application_name` | `appdevexp` | No | `terraform.tfvars` ✅ |
| `application_env` | `dev` | No | `terraform.tfvars` ✅ |
| `aws_account_region` | `us-west-1` | No | `terraform.tfvars` ✅ |
| `vpc_cidr` | `172.17.0.0/16` | No | `terraform.tfvars` ✅ |
| `az_count` | `2` | No | `terraform.tfvars` ✅ |
| `ecr_backend` | `580976914278.dkr.ecr...` | No¹ | `terraform.tfvars` ✅ |
| `ecr_frontend` | `580976914278.dkr.ecr...` | No¹ | `terraform.tfvars` ✅ |
| `domain` | `test.com` | No | `terraform.tfvars` ✅ |
| `enable_*` flags | `true` | No | `terraform.tfvars` ✅ |
| `db_username` | `mongoadmin` | **Yes** | `secrets.auto.tfvars` 🚫 |
| `db_password` | `hunter2` | **Yes** | `secrets.auto.tfvars` 🚫 |
| Any API key | `sk-...` | **Yes** | `secrets.auto.tfvars` 🚫 |

> ¹ ECR URIs contain the account ID (`580976914278`). Account IDs are considered low-sensitivity
> in AWS security guidance — IAM controls access, not URI obscurity. Committing ECR URIs is
> standard practice and required for Terraform to reference them.

#### Three-layer secret strategy

```
Layer 1 — Local development
  secrets.auto.tfvars    (gitignored, loaded automatically by terraform)
  Contains: db_username, db_password — any sensitive tfvar

Layer 2 — CI/CD (GitHub Actions)
  GitHub secret: DB_USERNAME → env var TF_VAR_db_username
  GitHub secret: DB_PASSWORD → env var TF_VAR_db_password
  Terraform reads TF_VAR_* automatically — no file needed

Layer 3 — Documentation
  terraform.tfvars.example  (committed, all variable names, no sensitive values)
```

### Step 3-1: Security groups — four groups, tight scope

```hcl
# ── Webapp ALB (external, internet-facing) ────────────────────────────────────
resource "aws_security_group" "webapp_alb" {
  name        = "${var.application_name}-webapp-alb-sg"
  description = "External ALB: HTTPS 443 from internet, egress to webapp tasks"
  vpc_id      = var.vpc_id

  ingress {
    description      = "HTTPS from internet"
    from_port        = 443
    to_port          = 443
    protocol         = "tcp"
    cidr_blocks      = ["0.0.0.0/0"]
    ipv6_cidr_blocks = ["::/0"]
  }

  egress {
    description = "Forward to webapp tasks within VPC"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = [var.vpc_cidr]
  }

  tags = merge(var.tags, { Name = "sg-webapp-alb" })
}

# ── Webapp tasks — ingress from external ALB, egress to API internal ALB ──────
resource "aws_security_group" "webapp_tasks" {
  name        = "${var.application_name}-webapp-tasks-sg"
  description = "Webapp tasks: port 3000 from webapp ALB; egress to API ALB and VPC endpoints"
  vpc_id      = var.vpc_id

  ingress {
    description     = "Next.js webapp from external ALB"
    from_port       = 3000
    to_port         = 3000
    protocol        = "tcp"
    security_groups = [aws_security_group.webapp_alb.id]
  }

  egress {
    description = "VPC egress: API internal ALB, Secrets Manager VPC endpoint, ECR"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = [var.vpc_cidr]
  }

  tags = merge(var.tags, { Name = "sg-webapp-tasks" })
}

# ── API ALB (internal, VPC-only) — accepts only from webapp tasks ─────────────
resource "aws_security_group" "api_alb" {
  name        = "${var.application_name}-api-alb-sg"
  description = "Internal API ALB: accepts HTTPS only from webapp tasks SG"
  vpc_id      = var.vpc_id

  ingress {
    description     = "HTTPS from webapp tasks only — no internet access"
    from_port       = 443
    to_port         = 443
    protocol        = "tcp"
    security_groups = [aws_security_group.webapp_tasks.id]
  }

  egress {
    description = "Forward to API tasks within VPC"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = [var.vpc_cidr]
  }

  tags = merge(var.tags, { Name = "sg-api-alb" })
}

# ── API tasks — ingress only from API ALB; MongoDB sidecar is localhost ───────
resource "aws_security_group" "api_tasks" {
  name        = "${var.application_name}-api-tasks-sg"
  description = "API tasks: port 3000 from API ALB SG only; no internet path"
  vpc_id      = var.vpc_id

  ingress {
    description     = "NestJS API from internal ALB only"
    from_port       = 3000
    to_port         = 3000
    protocol        = "tcp"
    security_groups = [aws_security_group.api_alb.id]
  }

  egress {
    description = "VPC egress: Secrets Manager, ECR, EFS mount targets"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = [var.vpc_cidr]
  }

  tags = merge(var.tags, { Name = "sg-api-tasks" })
}
```

> **Security property:** There is no path from the internet to the API task. The chain is:
> `Internet → sg-webapp-alb → sg-webapp-tasks → sg-api-alb → sg-api-tasks`. Each hop is
> a source-SG rule, not a CIDR range. Changing or removing any link in the chain
> immediately breaks internet access to the API without touching the API task itself.

**Conventional commit:**
```
infra(network): add four security groups for webapp ALB, webapp tasks, API ALB, and API tasks

- Webapp ALB: HTTPS 443 from internet; egress to VPC CIDR
- Webapp tasks: port 3000 from webapp ALB SG; egress to VPC CIDR
- API ALB: HTTPS 443 from webapp tasks SG only — no internet ingress
- API tasks: port 3000 from API ALB SG only — no internet path exists
```

### Container image reference table

| Container | docker-compose version | ECS image to use |
|---|---|---|
| MongoDB | `mongo:8.0-noble` | `mongo:8.0-noble` |
| PostgreSQL (tools DB) | `postgres:18-alpine` | `postgres:18-alpine` |
| Unleash | `unleashorg/unleash-server:8.0` | `unleashorg/unleash-server:8.0` |

| Prometheus | `prom/prometheus:v3` | `prom/prometheus:v3` |
| Keycloak | `quay.io/keycloak/keycloak:26.6` | `quay.io/keycloak/keycloak:26.6` |

> **Prometheus added:** It is part of `docker-compose.tools.yml` alongside Grafana. Grafana
> uses Prometheus as its primary data source (`depends_on: ob-prometheus`). It must be
> included to match the local dev environment.

> **Keycloak note:** `docker-compose.keycloak.yml` uses `start-dev --import-realm`. For ECS
> production mode use `start --optimized`. The env var names changed in Keycloak 26.x:
> `KC_BOOTSTRAP_ADMIN_USERNAME` / `KC_BOOTSTRAP_ADMIN_PASSWORD` (not `KEYCLOAK_ADMIN` / `KEYCLOAK_ADMIN_PASSWORD`).

> **Grafana-Keycloak OAuth:** Grafana uses Keycloak OIDC for SSO. In docker-compose, the
> `GF_AUTH_GENERIC_OAUTH_TOKEN_URL` uses the Keycloak container name. In ECS (awsvpc, same
> task), all containers share the network namespace, so `localhost` works for sidecar-to-sidecar
> calls. The `GF_SERVER_ROOT_URL` must be the public-facing Grafana URL (tools ALB domain).

> **Why tools cost doubled:** Prometheus requires headroom — tools task upgraded from

> adds ~$18/month.

> (single NAT GW, non-HA): saves ~$32/month → total ≈ **$313/month**.
